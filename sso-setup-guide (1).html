Azure Entra ID Single Sign-On — Setup Guide

End-to-end reference for enabling Entra ID SSO on the application. SSO is implemented entirely backend-side using the OAuth 2.0 Authorization Code flow with a confidential client; the frontend only knows the backend URL and reads a minted app JWT from the callback.

Status: SSO is not yet wired into the infrastructure — none of the five SSO env vars exist in any Bicep, pipeline, or parameter file today (only JWT_SECRET is wired). This guide covers both the Entra app registration and the steps to wire it up.
The one unbreakable rule: one account per user, seeded/created with the user's exact Azure AD UPN (the value in the id_token email / preferred_username). If the seeded email doesn't exactly match the real UPN, SSO creates a second, role-less account and the user fails every API call with 403 "You are not a member of department".

How it works

1
User clicks "Sign in with WTW SSO" — a full-page GET navigation (not fetch) to backend /auth/login.
2
Backend mints a 10-min state JWT (CSRF) and 302-redirects to the Entra authorize endpoint with scope openid profile email.
3
User authenticates at the Entra prompt.
4
Entra redirects back to backend /auth/azure/callback?code=&state=.
5
Backend re-validates state, then does a server-to-server token exchange (client_id + client_secret + code + redirect_uri).
6
Backend reads id_token claims (oid, email/preferred_username, name), upserts the user by OID, and mints the app's own HS256 app JWT.
7
Backend 302s to the frontend /login?token=<jwt>; the page reads the token, hydrates the profile, and stores the session.
8
Every API call sends Authorization: Bearer <jwt>; backend validates the HS256 token (Mode B) and loads the user by sub.
Coherence note: the callback always mints an HS256 app JWT, so the browser flow is coherent only with USE_SIMPLE_AUTH=true (Mode B). The Entra RS256 path (Mode C) validates Entra-issued tokens directly and would reject the broker-minted token. For this browser flow, keep USE_SIMPLE_AUTH=true.

A›Register the app in Entra ID
  1. New registration — name <APP_NAME>; account type single tenant (v2.0 endpoint).
  2. Add a Web platform (confidential client posts a secret — not SPA/public). Redirect URI: 
    https://<BACKEND_APP>.<CAE_DEFAULT_DOMAIN>/auth/azure/callback
    Local dev: http://localhost:8000/auth/azure/callback. The registered URI must exactly equal AZURE_AD_REDIRECT_URI.
  3. Create a client secret — record the value immediately (shown once) and note the expiry. This becomes AZURE_AD_CLIENT_SECRET.
  4. API permissions (Microsoft Graph, delegated): openid, profile, email, User.Read → then Grant admin consent for <TENANT> (org-wide silent sign-in).
  5. Token configuration — add the optional email claim to the ID token (not emitted by default in v2.0). Ensure users have a populated mail attribute; otherwise the code falls back to preferred_username (the UPN).
  6. Collect IDs from Overview: Directory (tenant) ID → ENTRA_TENANT_ID; Application (client) ID → ENTRA_CLIENT_ID.
Read the live CAE domain with the Azure CLI: 
az containerapp env show \
  --name <CONTAINER_APP_ENV> \
  --resource-group <RESOURCE_GROUP> \
  --query "properties.defaultDomain" -o tsv

C›Frontend configuration

The frontend has a single required variable:

Env varProd valueNotes
NEXT_PUBLIC_API_URLhttps://<BACKEND_APP>.<CAE_DEFAULT_DOMAIN>Absolute backend base URL, no trailing slash
Build-time, not runtime. Because it's prefixed NEXT_PUBLIC_, Next.js inlines it at next build into the client bundle. Set it in the build/CI environment; changing it requires a rebuild. A runtime-only container env var will not affect client reads — and if unset at build time, prod silently points at http://localhost:8000.

No client secrets, OAuth client IDs, or IdP config live in the frontend — SSO is entirely backend-driven. The callback delivers the token as /login?token=<jwt>; the page reads the token param, hydrates the profile, and stores the session locally.

D›Deployment wiring (the five SSO vars)

Container Apps never reads Key Vault directly. Each secret value flows through five layers:

KV secret → ADO variable group → @secure() Bicep param → Container App secret → secretRef env var

Add these five vars, keeping every layer in sync

ENTRA_TENANT_ID, ENTRA_CLIENT_ID, AZURE_AD_CLIENT_SECRET, AZURE_AD_REDIRECT_URI, FRONTEND_BASE_URL.

  1. secrets.bicep — new @secure() params + Key Vault secret resources (e.g. ENTRA-TENANT-ID, AZURE-AD-CLIENT-SECRET…). Bump the secret-count guard in the secrets pipeline.
  2. main.bicep — @secure() params + pass-through to the backend app.
  3. modules/backend.bicep — @secure() params + secrets[] entries + env[] secretRef entries.
  4. secrets paramfile step — new $(ENTRA-TENANT-ID) etc. mappings.
  5. secrets parameters file — values via the KV-backed variable-group path (do not commit the client secret in cleartext).
  6. Key Vault <KEY_VAULT> + variable group <SECRETS_VAR_GROUP>.
Cleartext warning: the secrets parameters file may already commit plaintext secrets. Add AZURE_AD_CLIENT_SECRET via the KV-backed flow only — never in cleartext in git.

Bake the frontend URL at build time

DOMAIN=$(az containerapp env show --name <CONTAINER_APP_ENV> \
  --resource-group <RESOURCE_GROUP> --query "properties.defaultDomain" -o tsv)
BACKEND_URL="https://<BACKEND_APP>.$DOMAIN"
az acr build ... --build-arg "NEXT_PUBLIC_API_URL=$BACKEND_URL" ... frontend
Redirect-URI rule: the Entra app registration's redirect URI must exactly equal both the deployed callback URL and the backend's AZURE_AD_REDIRECT_URI (sent in both the authorize request and the token exchange). Use the raw azurecontainerapps.io FQDN until a custom domain/cert is added.
E›Provisioning users & access

Roles (per-department)

RoleEffective access
SUBMITTERCreate runs, view own runs, export
REVIEWERView all dept runs, decide/comment, export, alias-approve (+ create runs)
DEPT_ADMINAll permissions

When a user has multiple role rows, the highest wins: DEPT_ADMIN > REVIEWER > SUBMITTER. A global is_platform_admin flag overrides as DEPT_ADMIN everywhere. Row-level security means REVIEWER/DEPT_ADMIN see all dept runs; SUBMITTER sees only their own.

Two ways to provision

  • Admin portal — the dept-admin user-management screen (create/re-add a user, set/replace role).
  • Seed migration — insert user + role rows; seeded rows start with entra_object_id = NULL and are claimed on first SSO sign-in.
On first SSO: a seeded row is claimed only when entra_object_id IS NULL AND its email equals the id_token email (case-insensitive, otherwise exact). No match → a brand-new row with zero memberships → 403 not a member. Seed the user's exact UPN.

Run migrations (manual, out-of-band)

There is no migration pipeline — run them as a one-shot Container App Job using the backend image:

az containerapp job create --name <MIGRATE_JOB> \
  --resource-group <RESOURCE_GROUP> \
  --image <ACR_NAME>.azurecr.io/<BACKEND_IMAGE>:<TAG> \
  --replica-timeout 300 \
  --command "python" "run_migrations.py" \
  --env-vars DB_SERVER=<SQL_SERVER> DB_NAME=<DB_NAME> ...
F›Verify & smoke test
  1. Backend env set: SKIP_AUTH=false, USE_SIMPLE_AUTH=true, strong JWT_SECRET, all four Entra values + FRONTEND_BASE_URL.
  2. Frontend bundle built with the correct NEXT_PUBLIC_API_URL (rebuild if changed).
  3. Test user has a row seeded with their exact UPN and an appropriate role.
  4. Open /login → click "Sign in with WTW SSO" → authenticate.

Success looks like: browser lands on the role-appropriate page; local session stores a valid access_token, email, role, dept; calls to /me, /me/departments return 200 with the Bearer header.

Troubleshooting

SymptomCause → Fix
403 "not a member of department"UPN ≠ seeded email → correct the seeded row to the exact UPN (or grant the role to the created row). One account per user.
503 from /auth/*One of the three Entra values is empty → set all three.
400 "invalid state" / loopState JWT failed — clock skew or JWT_SECRET changed mid-flow → stable secret, retry from clean /login.
502 "oid/email claim missing"id_token lacked claim → configure the optional email claim; ensure mail attribute or rely on preferred_username.
Redirect URI mismatch at EntraRegistered URI ≠ AZURE_AD_REDIRECT_URI → make both exactly equal.
Frontend hits localhost:8000 in prodNEXT_PUBLIC_API_URL unset at build → rebuild image with the build-arg.
Startup ValueError (Entra IDs required)Mode C engaged without Entra IDs → set them, or keep USE_SIMPLE_AUTH=true.
G›Security notes
  • Rotate the client secret before expiry: create a new Entra secret → update KV / variable group → redeploy/restart backend → remove the old one. KV-backed path only.
  • Override JWT_SECRET: the shipped dev default is public in source — if not overridden, app JWTs are forgeable by anyone who reads the repo. Strong, unique per environment.
  • Token in query string: the callback returns the app JWT as /login?token=..., which can leak via history, Referer, and access logs. Consider a one-time code or secure cookie as future hardening.
  • Verify the id_token signature (TODO): the callback currently decodes the id_token without re-verifying the RS256 signature/issuer/audience/expiry (it relies on the server-to-server TLS exchange). Recommend verifying against the Entra JWKS and checking aud/iss/exp.
